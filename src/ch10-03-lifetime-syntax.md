## 라이프타임으로 참조 검증하기

라이프타임은 우리가 이미 사용해 온 또 다른 종류의 제네릭입니다. 라이프타임은
타입이 원하는 동작을 갖도록 하는 것이 아니라, 참조가 우리가 필요로 하는 기간
동안 유효하도록 보장합니다.

4장의 [“참조와 빌리기”][references-and-borrowing]<!-- ignore --> 절에서 다루지
않은 한 가지 세부 사항이 있는데, 러스트의 모든 참조는 _라이프타임_ 을 가진다는
점입니다. 라이프타임은 그 참조가 유효한 스코프를 말합니다. 대부분의 경우, 대부분
타입이 추론되듯이 라이프타임도 암시적이고 추론됩니다. 여러 타입이 가능할 때만
타입을 명시해야 하듯이, 참조의 라이프타임이 여러 방식으로 관련될 수 있을 때만
라이프타임을 명시해야 합니다. 러스트는 제네릭 라이프타임 매개변수를 사용해 그
관계를 명시하도록 요구하여, 런타임에 사용되는 실제 참조가 확실히 유효하도록
보장합니다.

라이프타임을 명시한다는 개념은 대부분의 다른 프로그래밍 언어에는 없는 것이므로
낯설게 느껴질 것입니다. 이 장에서 라이프타임을 전부 다루지는 않겠지만, 여러분이
그 개념에 익숙해질 수 있도록 라이프타임 문법을 접할 일이 많은 일반적인 상황을
다룹니다.

### 라이프타임 명시 문법

라이프타임 명시는 참조들이 얼마나 오래 사는지에는 영향을 주지 않습니다. 오히려
여러 참조의 라이프타임이 서로 어떻게 관련되는지를, 라이프타임 자체에는 영향을
주지 않은 채 기술합니다. 함수가 시그니처에 제네릭 타입 매개변수를 지정해 어떤
타입이든 받을 수 있듯이, 제네릭 라이프타임 매개변수를 지정하면 어떤 라이프타임의
참조든 받을 수 있습니다.

라이프타임 명시는 다소 특이한 문법을 갖고 있습니다. 라이프타임 매개변수의
이름은 어퍼스트로피(`'`)로 시작해야 하며, 제네릭 타입처럼 보통 모두 소문자이고
매우 짧습니다. 대부분은 첫 번째 라이프타임 명시로 `'a`라는 이름을 사용합니다.
라이프타임 매개변수 명시는 참조의 `&` 뒤에 두며, 공백으로 명시와 참조 타입을
구분합니다.

다음은 예시입니다 — 라이프타임 매개변수 없는 `i32`에 대한 참조, 라이프타임
매개변수가 `'a`인 `i32`에 대한 참조, 라이프타임 `'a`를 가진 가변 `i32` 참조:

```rust,ignore
&i32        // a reference
&'a i32     // a reference with an explicit lifetime
&'a mut i32 // a mutable reference with an explicit lifetime
```

라이프타임 명시 하나만으로는 큰 의미가 없습니다. 명시는 여러 참조의 제네릭
라이프타임 매개변수가 서로 어떻게 관련되는지를 러스트에 알리기 위한 것이기
때문입니다. 이제 `longest` 함수의 맥락에서 라이프타임 명시가 서로 어떻게
관련되는지 살펴봅시다.

최소한 라이프타임 `'a`만큼 살아 있을 것임도 알려 줍니다. 실제로 이는 `longest`
함수가 반환하는 참조의 라이프타임이 함수 인수가 참조하는 값들의 라이프타임 중
더 작은 쪽과 같다는 것을 의미합니다. 이러한 관계가 바로 이 코드를 분석할 때
러스트가 사용하기를 우리가 원하는 것입니다.

이 함수 시그니처에 라이프타임 매개변수를 지정한다고 해서 전달되거나 반환되는 값들의
라이프타임이 바뀌는 것은 아니라는 점을 기억하세요. 오히려 이 제약을 따르지 않는
값은 빌림 검사기가 거부하도록 명시하는 것입니다. `longest` 함수가 `x`와 `y`가
정확히 얼마나 오래 사는지 알 필요는 없고, 이 시그니처를 만족하는 어떤 스코프가
`'a` 자리에 올 수 있음만 알면 됩니다.

함수에서 라이프타임을 명시할 때는 명시가 함수 본문이 아닌 시그니처에 놓입니다.
라이프타임 명시는 시그니처의 타입과 마찬가지로 함수의 계약의 일부가 됩니다.
함수 시그니처가 라이프타임 계약을 담고 있다는 것은 러스트 컴파일러가 수행하는
분석이 더 단순해질 수 있음을 의미합니다. 함수의 명시 방식이나 호출 방식에 문제가
있다면, 컴파일러 오류가 우리 코드와 제약의 어느 부분에서 문제가 생겼는지 더
정확하게 짚어 줄 수 있습니다. 만약 러스트 컴파일러가 우리가 의도한 라이프타임의
관계에 대해 더 많은 추론을 하도록 했다면, 컴파일러가 문제의 원인에서 여러 단계
떨어진 코드 사용 지점만 가리킬 수 있었을 것입니다.

### 라이프타임의 관계

라이프타임 매개변수를 지정하는 방식은 함수가 무엇을 하느냐에 따라 달라집니다.
예를 들어 `longest` 함수의 구현을 가장 긴 문자열 슬라이스가 아니라 항상 첫
번째 매개변수를 반환하도록 바꿨다면, `y` 매개변수에는 라이프타임을 지정할 필요가
없습니다. 다음 코드는 컴파일됩니다.

<Listing file-name="src/main.rs">

라이프타임은 `x`의 라이프타임이나 반환 값의 라이프타임과 아무 관계가 없으므로

그러나 그 값은 함수가 끝날 때 스코프를 벗어나므로, 이는 댕글링 참조가 됩니다.

여러분은 모든 참조가 라이프타임을 가지며, 참조를 사용하는 함수나 구조체에는
라이프타임 매개변수를 지정해야 한다고 배웠습니다. 그러나 Listing 4-9에는
라이프타임 명시 없이도 컴파일된 함수가 있었고, Listing 10-25에서 다시 보입니다.

<Listing number="10-25" file-name="src/lib.rs" caption="매개변수와 반환 타입이 참조임에도 라이프타임 명시 없이 컴파일된, Listing 4-9에서 정의한 함수">

```rust
{{#rustdoc_include ../listings/ch10-generic-types-traits-and-lifetimes/listing-10-25/src/main.rs:here}}
```

</Listing>

이 함수가 라이프타임 명시 없이 컴파일되는 이유는 역사적인 것입니다. 러스트
초기 버전(1.0 이전)에서는 모든 참조에 명시적 라이프타임이 필요했기 때문에
이 코드는 컴파일되지 않았습니다. 그때 함수 시그니처는 다음과 같이 작성되었을
것입니다.

```rust,ignore
fn first_word<'a>(s: &'a str) -> &'a str {
```

많은 러스트 코드를 작성한 후, 러스트 팀은 러스트 프로그래머가 특정 상황에서
같은 라이프타임 명시를 계속 반복해서 입력한다는 것을 발견했습니다. 이러한
상황은 예측 가능했고 몇 가지 결정론적 패턴을 따랐습니다. 개발자들은 이
패턴들을 컴파일러 코드에 프로그래밍해 두어, 그러한 상황에서는 빌림 검사기가
라이프타임을 추론할 수 있게 해 명시적 명시가 필요하지 않도록 했습니다.

이 러스트 역사가 관련 있는 이유는, 앞으로 더 많은 결정론적 패턴이 나타나
컴파일러에 추가될 가능성이 있기 때문입니다. 앞으로는 라이프타임 명시가 더
적게 필요할지도 모릅니다.

러스트가 참조를 분석하는 데 프로그래밍된 패턴을 _라이프타임 생략 규칙
(lifetime elision rules)_ 이라고 합니다. 이것은 프로그래머가 따라야 할
규칙이 아니라, 컴파일러가 고려할 특정 경우의 집합이며 우리 코드가 이
경우에 들어맞으면 라이프타임을 명시적으로 작성할 필요가 없습니다.

생략 규칙은 완전한 추론을 제공하지는 않습니다. 러스트가 규칙을 적용한 후에도
참조의 라이프타임이 무엇인지에 대한 모호함이 남아 있다면, 컴파일러는 남은
참조의 라이프타임이 무엇이어야 할지 추측하지 않습니다. 추측하는 대신, 우리가
라이프타임 명시를 추가해 해결할 수 있도록 오류를 내줍니다.

함수나 메서드 매개변수의 라이프타임은 _입력 라이프타임(input lifetimes)_,
반환 값의 라이프타임은 _출력 라이프타임(output lifetimes)_ 이라고 합니다.

컴파일러는 명시적 명시가 없을 때 참조의 라이프타임을 파악하기 위해 세 가지
규칙을 사용합니다. 첫 번째 규칙은 입력 라이프타임에, 두 번째와 세 번째
규칙은 출력 라이프타임에 적용됩니다. 컴파일러가 세 규칙의 끝에 도달했는데도
라이프타임을 알아낼 수 없는 참조가 남아 있다면, 컴파일러는 오류와 함께
중지됩니다. 이 규칙들은 `fn` 정의뿐 아니라 `impl` 블록에도 적용됩니다.

첫 번째 규칙은 컴파일러가 참조인 각 매개변수에 라이프타임 매개변수를 할당
한다는 것입니다. 다시 말해 매개변수가 하나인 함수는 하나의 라이프타임
매개변수를 얻습니다: `fn foo<'a>(x: &'a i32)`. 매개변수가 둘인 함수는 두 개의
개별 라이프타임 매개변수를 얻습니다: `fn foo<'a, 'b>(x: &'a i32, y: &'b i32)`.
그 이상도 마찬가지입니다.

두 번째 규칙은 입력 라이프타임 매개변수가 정확히 하나라면 그 라이프타임이
모든 출력 라이프타임 매개변수에 할당된다는 것입니다: `fn foo<'a>(x: &'a i32)
-> &'a i32`.

세 번째 규칙은 입력 라이프타임 매개변수가 여러 개이지만 그중 하나가 메서드
이기 때문에 `&self`나 `&mut self`라면, `self`의 라이프타임이 모든 출력
라이프타임 매개변수에 할당된다는 것입니다. 이 세 번째 규칙은 필요한 기호가
적어지므로 메서드를 훨씬 더 읽고 쓰기 좋게 만듭니다.

우리가 컴파일러라고 생각해 봅시다. 이 규칙들을 적용해서 Listing 10-25의
`first_word` 함수 시그니처에 있는 참조들의 라이프타임을 알아내 봅시다.
시그니처는 참조에 라이프타임이 붙지 않은 상태로 시작합니다.

```rust,ignore
fn first_word(s: &str) -> &str {
```

그런 다음 컴파일러는 첫 번째 규칙을 적용합니다. 이 규칙은 각 매개변수가 자기
라이프타임을 갖는다는 것입니다. 평소처럼 이름을 `'a`라고 하면, 시그니처는
이제 다음과 같습니다.

```rust,ignore
fn first_word<'a>(s: &'a str) -> &str {
```

입력 라이프타임이 정확히 하나이므로 두 번째 규칙이 적용됩니다. 두 번째 규칙은
하나의 입력 매개변수의 라이프타임이 출력 라이프타임에 할당된다는 것이므로,
시그니처는 이제 다음과 같습니다.

```rust,ignore
fn first_word<'a>(s: &'a str) -> &'a str {
```

이제 이 함수 시그니처의 모든 참조에 라이프타임이 있으므로, 컴파일러는
프로그래머가 이 함수 시그니처에서 라이프타임을 명시하지 않아도 분석을 계속
진행할 수 있습니다.

다른 예제를 살펴봅시다. 이번에는 Listing 10-20에서 처음 다룰 때 라이프타임
매개변수가 없었던 `longest` 함수입니다.

```rust,ignore
fn longest(x: &str, y: &str) -> &str {
```

첫 번째 규칙을 적용해 봅시다: 각 매개변수가 자기 라이프타임을 갖습니다. 이번에는
매개변수가 하나가 아니라 둘이므로 두 개의 라이프타임이 있습니다.

```rust,ignore
fn longest<'a, 'b>(x: &'a str, y: &'b str) -> &str {
```

입력 라이프타임이 하나 이상이므로 두 번째 규칙은 적용되지 않음을 볼 수 있습니다.
`longest`가 메서드가 아니라 함수이므로 매개변수 중 어느 것도 `self`가 아니고,
따라서 세 번째 규칙도 적용되지 않습니다. 세 규칙을 모두 적용한 뒤에도 반환
타입의 라이프타임이 무엇인지 알아내지 못했습니다. 이것이 Listing 10-20의
코드를 컴파일하려 할 때 오류가 났던 이유입니다. 컴파일러는 라이프타임 생략
규칙을 전부 적용했지만, 시그니처에 있는 모든 참조의 라이프타임을 여전히
알아낼 수 없었던 것입니다.

세 번째 규칙은 실제로 메서드 시그니처에서만 적용되므로, 메서드 시그니처에서
라이프타임을 명시할 일이 왜 자주 없는지 알아보기 위해 이어서 그 맥락에서의
라이프타임을 살펴봅시다.

### 메서드 정의에서 사용하기

라이프타임이 있는 구조체에 메서드를 구현할 때는, Listing 10-11에서 본 것처럼
제네릭 타입 매개변수와 같은 문법을 사용합니다. 라이프타임 매개변수를 어디서
선언하고 사용할지는, 그것이 구조체 필드와 관련이 있는지, 메서드 매개변수와
반환 값에 관련 있는지에 따라 달라집니다.

구조체 필드의 라이프타임 이름은 라이프타임이 구조체 타입의 일부이므로 항상
`impl` 키워드 뒤에 선언하고 구조체 이름 뒤에 사용해야 합니다.

`impl` 블록 안의 메서드 시그니처에서, 참조는 구조체 필드의 참조 라이프타임과
묶일 수도 있고 독립적일 수도 있습니다. 또한 라이프타임 생략 규칙 덕분에
메서드 시그니처에서는 라이프타임 명시가 필요 없는 경우가 많습니다. Listing
10-24에서 정의한 `ImportantExcerpt` 구조체를 사용한 몇 가지 예를 살펴봅시다.

먼저 유일한 매개변수가 `self`에 대한 참조이고 반환 값이 아무것도 참조하지
않는 `i32`인 `level`이라는 메서드를 사용해 봅시다.

```rust

```

`impl` 뒤의 라이프타임 매개변수 선언과 타입 이름 뒤의 사용은 필수지만, 첫
번째 생략 규칙 덕분에 `self`에 대한 참조의 라이프타임을 명시할 필요는 없습니다.

다음은 세 번째 라이프타임 생략 규칙이 적용되는 예입니다.

`&self`와 `announcement`에 각각의 라이프타임을 부여합니다. 그런 다음 매개변수
중 하나가 `&self`이므로, 반환 타입은 `&self`의 라이프타임을 얻고, 모든
라이프타임이 다뤄집니다.

### 정적 라이프타임

논의해야 할 특별한 라이프타임 하나가 `'static`으로, 해당 참조가 프로그램 전체
기간 동안 살아 있을 _수_ 있음을 나타냅니다. 모든 문자열 리터럴은 `'static`
라이프타임을 가지며, 다음과 같이 명시할 수 있습니다.

```rust
let s: &'static str = "I have a static lifetime.";
```

이 문자열의 텍스트는 프로그램의 바이너리에 직접 저장되며, 항상 사용 가능합니다.
따라서 모든 문자열 리터럴의 라이프타임은 `'static`입니다.

오류 메시지에서 `'static` 라이프타임을 사용하라는 제안을 볼 수도 있습니다.
그러나 참조의 라이프타임으로 `'static`을 지정하기 전에, 가지고 있는 참조가
프로그램의 전체 라이프타임 동안 정말 살아 있는지, 그리고 그렇기를 원하는지
먼저 생각해 보세요. 대부분의 경우 `'static` 라이프타임을 제안하는 오류 메시지는
댕글링 참조를 만들려는 시도나 사용 가능한 라이프타임 간 불일치에서 비롯됩니다.
그런 경우 해법은 그 문제들을 고치는 것이지, `'static` 라이프타임을 지정하는
것이 아닙니다.

영향을 주지 않습니다!
